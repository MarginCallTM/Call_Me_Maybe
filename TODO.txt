            Function Calling in LLMs  (constrained decoding "à la main")

But du projet : transformer un prompt en langage naturel
  ex: "What is the sum of 2 and 3?"
en un appel de fonction structuré et 100% valide :
  { "prompt": "...", "name": "fn_add_numbers", "parameters": {"a": 2.0, "b": 3.0} }

Principe central : on NE fait PAS confiance au modele pour produire du bon JSON.
On genere le JSON token par token et, a chaque etape, on met a -inf les logits
des tokens interdits (constrained decoding). Le JSON est donc valide PAR
CONSTRUCTION, pas par chance.

           - encode() -> Tensor -> .tolist()[0] -> input_ids
           - get_logits_from_input_ids(input_ids) -> 151936 floats
           - argmax manuel (greedy) -> next token id
           - vocab.json (id_to_token) -> token string
           Lance avec succes le 2026-06-05. Sortie observee :
             prompt "What is the sum of 2 and 3?" -> 11 tokens
             vocab size = 151936  (> 151643 entrees vocab.json : tokens speciaux + padding)
             argmax = 3555 'GWhat' (logit 17.38)
             top5 : GWhat, GTo, GAlso, GCC, GHow
[x] 1.3  Ouvrir vocab.json et comprendre sa structure :
           - mapping token_string -> id (a inverser : id -> token_string)
           - le symbole G (U+0120) = espace precedant un mot (ex: 'Gis', 'Gsum')
           - les chiffres sont des tokens individuels (ex: '2'->17, '3'->18)
           - l'espace seul est le token 220 ('G')
           - vocab.json a 151643 entrees < 151936 logits -> les IDs manquants
             sont des tokens speciaux/padding -> a masquer a -inf dans le decoder
[x] 1.4  Noter quels IDs correspondent aux caracteres cles du JSON :
           Exercice de masquage valide le 2026-06-05 : en mettant a -inf tous
           les logits sauf les chiffres, le modele est FORCE de sortir un
           chiffre ('1') au lieu de 'GWhat'. LECON : le masquage garantit la
           STRUCTURE (100% valide), pas le CONTENU (justesse vient du modele).
           IDs des tokens 1-caractere (vocab Qwen3-0.6B) :
             { =90  } =92  [ =58  ] =60  " =1  : =25  , =11  . =13  - =12
             chiffres 0-9 = 15..24 (bloc contigu)  true=1866  false=3849
             espace nu ' ' = None (l'espace n'existe que colle : 'G...')
           SUBTILITE MAJEURE -> tokens COMBINES de plusieurs chars :
             {" =4913   ": =788   ":" =3252   ", =497   }, =2137   "} =9207
             => "1 char = 1 token" est FAUX. Le decodeur (Phase 5) devra gerer
                qu'un token avance le JSON de plusieurs caracteres.
           DECISIONS DE DESIGN retenues :
             - generer du JSON COMPACT sans espaces ({"a":2.0}) pour eviter le G
             - NE PAS hardcoder les IDs : les recalculer au runtime depuis vocab.json

PHASE 5 — LE COEUR : CONSTRAINED DECODING -> src/constrained_decoder.py
================================================================================
[ ] 5.1  Concevoir la machine a etats du JSON attendu. Etats typiques :
           START -> "{" -> attendre cle -> ":" -> valeur -> "," ou "}" -> END
         Adapter selon le schema de la fonction choisie.
[ ] 5.2  Pour chaque etat, definir l'ENSEMBLE des tokens autorises
           (ex: apres ":" pour un parametre "number" -> chiffres, "-", ".")
[ ] 5.3  Boucle de generation token par token :
           a) recuperer les logits via get_logits_from_input_ids(list[int]) -> list[float]
           b) calculer le masque des tokens autorises (selon l'etat)
           c) mettre les logits interdits a -inf (liste Python suffit, numpy optionnel)
              ex: logits[i] = float('-inf')  OU  np.array(logits)[mask] = -inf
           d) prendre l'argmax PARMI les tokens autorises (index = prochain token ID)
           e) ajouter ce token ID a la sequence, mettre a jour l'etat
[ ] 5.4  Contrainte de TYPE selon functions_definition.json :
           - number : entier ou flottant valides
           - string : ouverture/fermeture de guillemets, echappements
           - boolean : true / false
[ ] 5.5  Condition d'arret : JSON complet (toutes les cles requises remplies)
[ ] 5.6  Garde-fou : limite max de tokens pour eviter une boucle infinie
[ ] 5.7  Verifier que la sortie est TOUJOURS du JSON valide et conforme au schema
